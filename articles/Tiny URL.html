System Design — Tiny URL

A production-grade URL shortening service capable of handling billions of redirects at sub-10ms latency with global availability and intelligent analytics.

Published April 17, 2025
Read Time 45 min
Type HLD + LLD
Level Senior / Architect
01

Document Overview

This System Design Document (SDD) provides comprehensive technical guidance for building a globally distributed, highly available URL shortening service — commonly referenced as TinyURL. It covers both High-Level Design (HLD) and Low-Level Design (LLD), enabling engineering teams to implement, scale, and operate the system end to end.

Scope

In Scope:

  • URL shortening and alias generation engine
  • Redirect service with sub-10ms latency
  • Custom alias support and expiry policies
  • Click analytics and geo-tracking
  • User authentication, API key management
  • Admin dashboard and rate limiting
  • Multi-region active-active deployment

Out of Scope:

  • Payment processing or monetisation layer
  • Content moderation (separate ML pipeline)
  • Native mobile applications

Audience

Backend Engineers, Platform Architects, Tech Leads, SRE/DevOps Teams, and System Design Interview Panels.

System Overview

System Objectives

  • Generate globally unique short URLs in under 50ms with zero collisions
  • Serve redirect responses with p99 latency below 10ms at global scale
  • Support 10 billion total stored URLs with horizontal scaling
  • Handle 100,000 redirect RPS at peak with burst tolerance of 3x
  • Provide real-time click analytics with 1-second event delivery lag
  • Offer 99.99% uptime SLA via active-active multi-region topology
  • Enforce per-user rate limits and per-URL expiry policies reliably
  • Comply with GDPR for EU user data and CCPA for California residents

System Context

TinyURL sits at the intersection of a user-facing API platform, a real-time analytics pipeline, and a global redirect delivery network. It integrates with identity providers, CDN edge networks, and downstream data warehouses for business intelligence.

Architectural Style

Decision — Microservices with Event-Driven Analytics

Chosen: Domain-separated microservices (Redirect, Shorten, User, Analytics) communicating via REST internally and Kafka for async analytics events.

Alternatives Considered: Monolith (poor independent scaling), Lambda/Serverless (cold-start kills latency SLA), GraphQL gateway (over-engineered for simple redirect).

Selection Criteria: Redirect service must scale independently from write path. Analytics must not block the critical redirect path. Operational isolation reduces blast radius.

Trade-offs: Adds operational complexity vs monolith; mitigated by Kubernetes service mesh and standardised deployment templates.

Key Design Principles

  • Single Responsibility: Each microservice owns one domain; Redirect Service never writes URL metadata.
  • Eventual Consistency (CAP — AP): Redirect reads tolerate stale cache; creation path is strongly consistent via DynamoDB conditional writes.
  • DRY Configuration: All service configs centralised in AWS Parameter Store; no env-specific hardcoding.
  • Fail Fast: Timeouts and circuit breakers at every inter-service call boundary with fallback to direct DB reads.
  • Idempotency: URL creation API is idempotent — duplicate long URL returns existing short alias.
  • Zero-Trust Networking: mTLS between all internal services; no service-to-service HTTP without client certificate.
  • Observability-First: Distributed tracing injected at entry point; every service emits structured JSON logs and Prometheus metrics.
  • Defence in Depth: WAF at CDN edge, rate limiting at API Gateway, input validation at application layer, DB access via IAM roles only.

Component Design

Redirect Service (Go): The most latency-sensitive component. Receives slug, checks Redis L1 cache, falls back to DynamoDB on miss, emits analytics event to Kafka, returns HTTP redirect. No write operations. Stateless; scales horizontally to 200 pods.

Shorten Service (Go): Handles URL creation. Generates or validates alias via Snowflake-inspired ID → Base62. Writes to DynamoDB with conditional put (prevents collision). Warms Redis cache. Calls Google Safe Browsing API synchronously before persisting.

User Service (Node.js): Manages user registration, OAuth2 flows, API key issuance (UUID v4 hashed with SHA-256), quota enforcement. Reads from PostgreSQL. Publishes audit events to Kafka.

Analytics Ingestor (Go): Consumes click events from Redirect Service via Kafka. Runs Flink stateful aggregations for windowed click counts. Writes enriched events (with GeoIP) to BigQuery and increments Redis counters for real-time dashboard.

Admin Service (Node.js): Internal-only service behind VPN. Provides REST API for URL deactivation, user management, abuse reporting. Writes directly to DynamoDB and PostgreSQL with operator-level IAM roles.

State Management Strategy

URL records are persisted in DynamoDB (source of truth) with a Redis write-through cache for hot redirects. Session state is stateless (JWT bearer tokens). Distributed rate limiting uses Redis atomic INCR with sliding window. Analytics state is managed by Flink with RocksDB backend for fault-tolerant checkpointing.

Consistency Model

The system applies a nuanced consistency strategy by partition: URL creation uses DynamoDB conditional writes for strong consistency (preventing collision). Redirect reads use Redis with eventual consistency — stale cache is tolerable for milliseconds. Click analytics accepts eventual consistency with Kafka at-least-once delivery and Flink idempotent aggregation. This balances the CAP theorem trade-offs: the system is AP for reads and CP for writes.

Data Security

  • Encryption at rest: DynamoDB AWS-managed CMK (KMS); PostgreSQL Aurora encrypted volumes; Kafka MSK TLS encryption.
  • Encryption in transit: TLS 1.3 enforced end-to-end; mTLS between microservices via Istio service mesh.
  • PII masking: IP addresses in click events hashed with HMAC-SHA256 before DW persistence for GDPR compliance.
  • Access control: All DB access via IAM roles — no static credentials. Least-privilege policies per service.
  • GDPR compliance: EU-resident click event data isolated to eu-west-1; right-to-erasure endpoint deletes URL records and associated click events within 72 hours.

Integration Architecture

Integration Patterns

  • REST (HTTP/JSON): Primary API contract for URL creation, user management, and admin operations. Versioned under /api/v1/ with OpenAPI 3.1 spec.
  • Kafka Event Streaming: Async decoupling of click analytics from the redirect hot path. Zero latency impact on redirect SLA.
  • Webhooks: Outbound HTTP callbacks to user-registered endpoints on bulk job completion and URL expiry events.
  • Google Safe Browsing API: Synchronous HTTP call at URL creation time; cached 24h per URL hash to reduce quota usage.
  • MaxMind GeoIP2: Embedded database (updated weekly via S3) for IP-to-country resolution in the analytics ingestor — no external call per event.

API Design Standards

  • Versioning: URI-based versioning (/api/v1/); breaking changes bump major version.
  • Naming: Plural nouns for resources (/urls, /users); kebab-case for multi-word paths.
  • Request/Response: JSON body; Content-Type: application/json; standard error envelope: {error, code, message, requestId}.
  • Rate Limiting: Response headers X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset on every response.
  • Idempotency: POST /api/v1/urls accepts Idempotency-Key header for safe retries.

Security Architecture

Security Principles

  • Defence in Depth: WAF → API Gateway → App Layer → DB — each layer enforces independent security controls.
  • Least Privilege: Each microservice has a dedicated IAM role with minimum DynamoDB/S3/Kafka permissions needed.
  • Zero Trust Networking: No implicit trust between services; mTLS via Istio service mesh for all pod-to-pod communication.
  • Fail Secure: On auth service degradation, requests are rejected (403) rather than allowed through.
  • Audit Everything: All mutation operations produce immutable audit logs to CloudWatch Logs with tamper detection.
  • Shift Left Security: SAST (Semgrep), SCA (Snyk), DAST (OWASP ZAP) integrated into CI pipeline before any merge.

Auth / Authz Model

Mechanism: OAuth2 Authorization Code Flow for web users; API Key (JWT RS256) for programmatic access. JWTs are verified against Auth0 JWKS endpoint cached locally for 24h.

RBAC Roles: user (create/read own URLs), admin (manage all URLs + users), readonly (analytics view only). Role claims embedded in JWT payload.

MFA: TOTP-based MFA enforced for admin role via Auth0 MFA policies. Optional for standard users.

Session Management: Stateless JWT (15-min access token, 7-day refresh token). Refresh token rotation with single-use enforcement stored in Redis.

Network Security

  • TLS/mTLS: TLS 1.3 at CloudFront; mTLS between all pods via Istio mutual authentication.
  • Segmentation: VPC with public subnet (ALB/CloudFront), private subnet (services), isolated subnet (DB). No direct DB access from public subnet.
  • DDoS Protection: AWS Shield Advanced on CloudFront; WAF rate-based rules auto-block IPs exceeding 10,000 req/min.
  • Firewall Rules: Security groups allow only port 443 inbound; inter-service traffic on pod CIDR only via Istio policy.

Application Security

  • Input Validation: URL format validated via RFC 3986 regex; max length 2,048 chars; scheme restricted to https/http.
  • OWASP Mitigations: Parameterised queries (no SQL injection surface); CORS policy restricted to registered origins; HSTS header enforced.
  • Dependency Scanning: Snyk in CI pipeline; critical CVEs block merge; weekly full-dependency audit.
  • SAST/DAST: Semgrep on every PR; OWASP ZAP nightly against staging environment.

Caching Strategy

  • Browser cache: HTTP 301 with Cache-Control: max-age=86400 for permanent URLs; 302 with no-cache for expiring URLs.
  • CDN (CloudFront): Caches redirect responses at 400+ edge PoPs globally. Cache TTL 3,600s for stable URLs. Purge via CloudFront invalidation on URL deactivation.
  • Application cache (Redis): Write-through on URL creation; TTL matches URL expiry. LRU eviction policy. Hit ratio target: 95%+. Warming: top 10,000 URLs pre-loaded on pod startup.
  • DB query cache: DynamoDB DAX considered but not used — Redis already serves this role with higher flexibility.

Scalability Design

  • Horizontal Scaling: Redirect Service pods scale via HPA on CPU > 60% or RPS > 500/pod. Max 200 pods per region.
  • Auto-scaling: DynamoDB on-demand mode adjusts RCU/WCU automatically; no capacity planning needed for DB layer.
  • Sharding: Redis cluster mode with 6 shards; consistent hashing distributes keys. Short aliases are hash keys — uniform distribution.
  • Read Replicas: PostgreSQL Aurora has 2 read replicas for user service queries; writes go to primary.

Query Optimisation

  • DynamoDB access pattern: Single-table design with short_alias as PK; no scans, only point reads. GetItem is O(1).
  • PostgreSQL indexes: B-tree index on users.email and api_keys.hashed_key for O(log n) lookups.
  • Connection pooling: PgBouncer sidecar in transaction mode; max 100 connections per Aurora instance.
  • Async processing: Analytics events fire-and-forget via Kafka; redirect response does not wait for Kafka ack.

High Availability Design

  • Redundancy: All services run minimum 3 pods across 3 AZs. ALB routes only to healthy pods (health check every 10s).
  • Failover: Route53 health-check-based latency routing; automatic DNS failover to secondary region within 60s.
  • Circuit Breakers: Hystrix/Resilience4j pattern: open after 5 consecutive DB failures; half-open probe every 30s.
  • Retry Logic: Exponential backoff with jitter (100ms base, 3 attempts) for DynamoDB; no retry on Kafka produce (fire-and-forget).
  • Health Checks: /healthz (liveness) and /readyz (readiness) on all pods; separate DB connectivity check on readyz.

Resilience Patterns

  • Bulkhead: Separate thread pools for DB calls vs Kafka produce vs external API calls; failure in one does not exhaust others.
  • Graceful Degradation: If Redis is unavailable, Redirect Service falls back to DynamoDB with 30ms latency penalty — no user-facing error.
  • Timeout Configuration: Redis: 5ms; DynamoDB: 50ms; Google Safe Browsing: 200ms; all configurable via Parameter Store.

Observability Architecture

Monitoring Strategy

All services emit Prometheus metrics on /metrics; scraped every 15s. Key SLIs (redirect latency, error rate, cache hit ratio) tracked as Grafana SLO dashboards. Alerting via Alertmanager → PagerDuty with severity tiers (P1–P4).

Logging Architecture

  • Format: Structured JSON logs with fields: timestamp, level, service, traceId, spanId, requestId, userId, message, durationMs.
  • Aggregation: Fluent Bit DaemonSet ships pod logs to Grafana Loki; label-indexed by service, env, region.
  • Retention: 30 days hot in Loki; 1 year cold in S3 Glacier via Loki ruler.
  • Correlation: traceId injected at API Gateway and propagated via W3C Trace Context headers across all services.

Distributed Tracing

  • Implementation: OpenTelemetry SDK in all services; OTLP export to Grafana Tempo.

  • Sampling: 100% sampling for errors and latency > 100ms; 1% head-based sampling for normal traffic.

Technology Decisions and Trade-offs

Decision 1 — DynamoDB over MySQL for URL Storage

Chosen: Amazon DynamoDB. Alternatives: MySQL/PostgreSQL, Cassandra, MongoDB.

Criteria: Single-digit ms reads at any scale; no schema migration overhead; auto-scaling RCU/WCU; native TTL for expiry.

Trade-offs: No native SQL joins (not needed — access pattern is always point read by PK). Higher cost per GB vs MySQL. Mitigated by single-table design maximising efficiency.

Decision 2 — Base62 Encoding over MD5 Hashing for Alias Generation

Chosen: Snowflake-style distributed counter → Base62 encode to 7 chars. Alternatives: MD5 hash truncation, UUID, Nano ID.

Criteria: Guaranteed zero collisions (counter is monotonically unique); compact output; no retry on collision needed.

Trade-offs: Counter service is a potential bottleneck. Mitigated by pre-allocating counter ranges (batch of 1000) per Shorten Service pod — no coordination needed per request.

Decision 3 — Redis over Memcached for Caching

Chosen: Redis (ElastiCache). Alternatives: Memcached, in-process LRU cache.

Criteria: TTL-native (aligned with URL expiry); cluster mode for HA; also serves rate-limiting counters (INCR + EXPIRE atomic ops); Lua scripting for sliding window.

Trade-offs: Slightly higher memory overhead vs Memcached. Mitigated by zipping cache values and using efficient Redis data types.

Decision 4 — Go over Java for Redirect Service

Chosen: Go 1.22. Alternatives: Java (Spring Boot), Rust, Python.

Criteria: Sub-ms goroutine context switch; tiny binary (distroless image <12MB); native concurrency model; standard library HTTP server handles 50k+ concurrent conns with minimal memory.

Trade-offs: Smaller talent pool than Java. Mitigated by team upskilling programme and Go's simplicity relative to JVM.

Decision 5 — HTTP 301 vs HTTP 302 Redirect

Chosen: 301 for permanent URLs (no expiry); 302 for expiring URLs. Trade-offs: 301 is cached by browsers indefinitely — analytics undercounts repeat clicks from same browser. Mitigated by tracking unique vs repeat clicks at CDN layer via cookies for analytics-sensitive customers. 302 forces every browser request through the redirect service, enabling full analytics at the cost of higher origin traffic.

Decision 6 — Kafka over SQS for Analytics Events

Chosen: Apache Kafka (AWS MSK). Alternatives: AWS SQS, AWS Kinesis, RabbitMQ.

Criteria: Consumer group replay for backfill; topic partitioning by alias for ordered per-URL events; Flink-native Kafka connector; 7-day retention without extra cost.

Trade-offs: Operational complexity vs SQS. Mitigated by using MSK Serverless — no broker management needed.

Senior Architect Interview Questions

Tier 1 — Architecture and Design

Q1. Why choose an active-active multi-region topology over active-passive for this system?
What a great answer covers:
  • Redirect latency benefit: active-active routes user to nearest region; active-passive adds 50–100ms cross-region hop for non-primary users
  • RTO difference: active-active = 0s (traffic already flowing); active-passive = 60s DNS failover
  • Data conflict handling: DynamoDB Global Tables uses last-writer-wins; URL creation is idempotent so conflicts are benign
  • Cost trade-off: active-active costs ~2× infra; justified by 99.999% SLA requirement for redirect service
  • GDPR constraint: EU region is isolated from US/APAC replication — nuances the "active-active everywhere" claim
Q2. How does your alias generation strategy guarantee zero collisions at 100k URLs/second without a centralised lock?
What a great answer covers:
  • Snowflake-inspired ID: worker_id (10 bits) + timestamp (41 bits) + sequence (12 bits) = globally unique 63-bit integer without coordination
  • Base62 encode to 7 chars: 62^7 = 3.5 trillion IDs — namespace exhaustion not a concern for 20+ years at current scale
  • DynamoDB conditional write as safety net: attribute_not_exists(short_alias) — even if two workers somehow generate the same ID (impossible with Snowflake), DB write fails cleanly
  • Counter range pre-allocation: each Shorten Service pod pre-allocates a range of 1,000 IDs from a shared Atomic Counter in DynamoDB — eliminates per-request network coordination
  • Explain trade-off vs MD5 hash: hash has collision probability and requires retry; counter is deterministic and collision-free
Q3. What are the single points of failure in this design and how are each mitigated?
What a great answer covers:
  • API Gateway (Kong): multi-pod deployment with ALB; stateless config from DB; health check routes around failed pod
  • Redis: ElastiCache cluster mode with auto-failover; Redirect Service degrades gracefully to DynamoDB on cache miss
  • DynamoDB Global Tables: managed; no operator-facing SPOF; AWS SLA 99.99%
  • DNS (Route53): AWS managed; health-check-based failover; low TTL (30s) for fast convergence
  • Kafka MSK: 3-AZ broker replication; consumer group offsets persist on Kafka itself; Flink checkpoints every 60s
  • Acknowledge that CloudFront edge node failure is theoretically possible — mitigated by 400+ PoP redundancy
Q4. Apply the CAP theorem to this system. Which guarantee do you sacrifice and where?
What a great answer covers:
  • Write path (URL creation): CP — DynamoDB conditional write enforces consistency; availability sacrifice is acceptable (creation can retry)
  • Read path (redirects): AP — eventual consistency via Redis cache; stale-but-functional redirects preferred over unavailability
  • Analytics: AP — Kafka at-least-once; click counts may temporarily lag; correctness over latency is not a hard requirement here
  • Network partition: with multi-region DynamoDB, partition in replication link means both regions serve potentially diverged data (AP) until reconciliation
  • Show maturity: CAP is a spectrum not a binary; PACELC model better captures the latency vs consistency trade-off in normal (non-partition) operation

Tier 2 — Scalability and Performance

Q5. A viral tweet causes redirect traffic to spike from 100k to 1M RPS in 30 seconds. Walk me through what happens in the system.
What a great answer covers:
  • CloudFront absorbs 80%+ of the spike from edge cache (301 cached at PoP); only ~200k RPS hits origin
  • HPA triggers on CPU > 60%; Karpenter provisions new EC2 nodes in ~45s — there is a 30–45s window of pressure
  • DynamoDB on-demand auto-scales RCU; no throttling expected; one viral URL = hot partition mitigated by DynamoDB Adaptive Capacity
  • Redis cluster handles read spike without saturation (single key = round-robin to one shard; ElastiCache replication lag <1ms)
  • Honest answer: the 30-second pre-scale gap means p99 may degrade to ~50ms during Karpenter warmup — CDN cache is the true spike absorber
  • Mitigation: predictive pre-scaling on Kafka click-velocity signal; CDN cache warming for top-100 trending URLs
Q6. How do you handle the thundering herd problem when Redis cache is cold (e.g. after a cluster restart)?
What a great answer covers:
  • Without mitigation: cache cold → all 100k RPS hit DynamoDB simultaneously → DynamoDB throttling → cascading failure
  • Solution 1: Cache warming on pod startup — Redirect Service reads top 10,000 most-clicked URLs from DynamoDB and pre-populates Redis before accepting traffic
  • Solution 2: Request coalescing (dogpile prevention) — use Redis SETNX to set a "lock" key; only one goroutine fetches from DynamoDB per cache miss per key; others wait 5ms and retry
  • Solution 3: Probabilistic early expiration — refresh cache slightly before TTL expires using rand() < β * TTL * -log(rand()) formula to spread renewal load
  • Solution 4: Staggered rollout — roll Redis cluster restart one shard at a time, not all at once
Q7. The analytics team wants sub-second real-time click counts. How does the current design support this and what would you change?
What a great answer covers:
  • Current design: Redis counter incremented atomically on each redirect (fire-and-forget INCR); dashboard reads Redis counter for near-real-time view (~1s lag)
  • Flink 1-second event-time windows aggregate Kafka events and flush to BigQuery — not suitable for sub-second; Redis is the answer for real-time
  • Enhancement: WebSocket push from a counter change notification service (Redis keyspace notifications → SSE to dashboard); eliminates polling
  • Trade-off: INCR in Redis per redirect adds ~0.1ms; at 100k RPS = 100k INCR/s on one key = hot write; mitigate with counter sharding (10 shards, sum on read)
  • Demonstrate depth: HyperLogLog for unique visitor count at massive scale (12 KB memory for 1B unique IPs with 0.81% error)
Q8. How would you identify and fix a performance bottleneck that only appears at 80k+ RPS?
What a great answer covers:
  • Observability-first: Grafana Tempo flame graph to find the slowest span at high RPS — typically connection pool exhaustion or GC pause
  • Common culprits: Redis connection pool exhaustion (default pool too small for 200 pods × concurrent requests); fix with pool size = max_concurrency / pod_count
  • DynamoDB batch get vs individual GetItem at scale; HTTP keep-alive tuning on Go HTTP client
  • Network saturation: check AWS ENI bandwidth limits per instance type; upgrade to m6i.4xlarge if hitting 12.5 Gbps limit
  • Process: reproduce in staging with Locust at 85k RPS → profile with Go pprof → fix → verify p99 improvement → release with canary

Tier 3 — Security and Reliability

Q9. A security researcher reports your URL shortener is being used to distribute malware. What is your incident response?
What a great answer covers:
  • Immediate (0–5 min): Admin Service deactivates flagged short aliases (sets is_active=false in DynamoDB); CloudFront invalidation removes cached 301; CDN edge now returns 410 Gone
  • Short-term (5–30 min): Query DynamoDB for all URLs sharing same long URL domain; bulk deactivate entire domain; add domain to WAF blocklist
  • Investigation: review creator's API key and account; suspend account; check if API key compromised or deliberate abuse
  • Prevention: retroactive Google Safe Browsing re-check on all URLs from that account; add to abuse report feed
  • Communication: notify security researcher; post incident report to status page within 4 hours; update abuse@ policy
  • Systemic fix: if Safe Browsing missed it, evaluate additional ML-based URL classifier and consider human review queue for high-risk domains
Q10. How do you ensure a full AWS region failure results in zero data loss for URL records?
What a great answer covers:
  • DynamoDB Global Tables: multi-master active-active replication; writes in us-east-1 replicated to eu-west-1 in sub-second; RPO ≈ 0
  • Route53 health check: detects us-east-1 ALB failure within 30s; DNS TTL 30s; traffic switches to eu-west-1 within 60s total
  • Redis: cache is warm-able from DynamoDB; losing Redis = 0 data loss (cache is derivative of source of truth)
  • Kafka MSK: cross-region MirrorMaker 2 replication to secondary region; analytics events may have up to 5s lag in replication — at-least-once delivery guarantees eventual consistency
  • PostgreSQL: Aurora Global Database with cross-region replica; RPO < 1s; RTO ~1 min for manual promotion to secondary
  • Demonstrate operational readiness: quarterly failover drill simulates region loss; playbook is practised not just documented
Q11. How would you protect this system from a volumetric DDoS attack targeting the redirect endpoint?
What a great answer covers:
  • Layer 7: AWS WAF rate-based rules on CloudFront — auto-block IPs exceeding 10,000 req/min within 5 minutes; no manual intervention
  • Layer 4/3: AWS Shield Advanced absorbs SYN floods and UDP reflection attacks at CloudFront edge; scrubbing capacity up to Tbps
  • Anycast routing: CloudFront distributes attack traffic across 400+ PoPs — no single origin is overwhelmed
  • Origin shielding: CloudFront Origin Shield means only one PoP acts as cache origin; attack traffic that reaches origin is reduced to cache-miss fraction only
  • API Gateway: Kong rate limiting is a second line of defence for authenticated API traffic; blocks per-key abuse even if WAF passes traffic
  • Cost concern: Shield Advanced fixed cost $3,000/month — justified by SLA value; DDoS cost protection reimburses AWS bill spikes

Tier 4 — Scenario-Based

Q12. At 2am on Black Friday, redirect error rate spikes to 5%. Walk me through your diagnosis and remediation steps.
What a great answer covers:
  • Step 1: Check Grafana SLO dashboard — is it all regions or one? Is it all aliases or specific ones?
  • Step 2: Open Grafana Tempo — find p99 traces for erroring requests; identify which span is failing (Redis? DynamoDB? routing?)
  • Step 3: Check recent deployments in ArgoCD — was there a release in the last 2 hours? If yes, rollback is fastest mitigation
  • Step 4: If no recent deploy — check CloudWatch for DynamoDB throttle count and Redis eviction rate; check pod OOM events (kubectl get events)
  • Step 5: If DynamoDB throttling, switch to DynamoDB on-demand mode immediately via Terraform apply (3-minute change)
  • Step 6: Declare P1 incident; update status page; assign incident commander; start 15-minute update cadence
  • Post-incident: blameless post-mortem within 48h; add load test scenario covering Black Friday traffic profile
Q13. You discover that 500,000 URL records were corrupted in DynamoDB due to a bug in the Shorten Service deployed last night. How do you recover?
What a great answer covers:
  • Immediate: rollback the buggy Shorten Service deployment via ArgoCD; no new corruptions after rollback
  • Scope assessment: query DynamoDB using GSI on created_at timestamp range for records created during the bad deploy window; export to S3 for analysis
  • Recovery: use DynamoDB PITR (Point-in-Time Recovery) to restore the table to T-1 hour (before deploy); replay valid writes from Kafka topic (URLs created legitimately in the window)
  • Kafka replay: Flink consumer replays URL creation events from the affected timestamp range; re-applies them to the restored table
  • Validation: reconcile record count and spot-check 1,000 random aliases; verify redirects resolve correctly
  • Communication: notify affected users if their URLs were impacted; personalised email from user_id in URL records
Q14. Finance wants to cut infrastructure costs by 40%. Which components would you target and what are the trade-offs?
What a great answer covers:
  • Compute (30% saving): right-size Redirect Service pods using VPA recommendations; use Spot Instances for Flink analytics (fault-tolerant workload); Karpenter consolidation reduces node count by 20%
  • DynamoDB (15% saving): switch analytics GSI queries to S3 Select on exported Parquet; remove expensive DynamoDB GSIs used only for analytics
  • Redis (10% saving): reduce Redis shard count from 6 to 4 by evicting less-popular URLs earlier (reduce TTL from 1h to 20min for URLs with zero clicks in 24h)
  • CloudFront (5% saving): increase cache TTL for stable URLs from 1h to 24h; reduces origin requests by additional 10%
  • What not to cut: Shield Advanced ($3k/month) is non-negotiable for DDoS protection; multi-region replication is non-negotiable for 99.99% SLA
  • Show maturity: cost cutting should be driven by FinOps dashboards (AWS Cost Explorer + Kubecost), not guesswork; measure before and after
