What is Kubernetes Ingress?

Kubernetes Ingress is a cluster-level API object that manages external HTTP and HTTPS access to services inside a cluster, providing URL-based routing, hostname routing, TLS termination, and load balancing — all from a single declarative configuration.

Simple analogy: Think of Ingress as a smart hotel receptionist. Guests (external traffic) all walk through one front door. The receptionist checks your name (hostname) and room request (URL path) and directs you to the right room (Service). Without a receptionist, every room would need its own dedicated entrance (LoadBalancer per Service).

Definition 1: An Ingress is a Kubernetes resource that declaratively defines rules for routing external HTTP(S) traffic to internal ClusterIP services, with optional TLS termination.

Definition 2: Ingress is the Kubernetes way to expose many web services through a single external IP, using host and path rules — without provisioning a cloud load balancer per service.

Category: Networking / Traffic Management / API Gateway Pattern

Official docs: kubernetes.io/docs/concepts/services-networking/ingress

2

Why Does It Exist?

Before Ingress, exposing a service externally meant creating a LoadBalancer type Service — which provisions a cloud load balancer (and bills you) per service. A cluster with 20 microservices meant 20 load balancers, 20 external IPs, and no centralised SSL management or URL routing. Ingress solves all of this with a single entry point.

Without IngressWith Ingress
1 LoadBalancer per Service = 20 cloud LBs for 20 services1 Ingress Controller handles all 20 services
No centralised TLS — each service manages its own certTLS terminated at the Ingress; cert-manager automates renewal
No URL path routing — each service has its own IP:portRoute /api → backend, /app → frontend, all on port 443
No hostname-based routingapp.example.com → app service, api.example.com → API service
High cost — each cloud LB billed separatelySingle LB fronting the Ingress Controller

History & Evolution

  • 2015Kubernetes 1.1 — Ingress introduced as a beta API. NGINX Ingress Controller was among the first implementations. Routing rules were simple host + path → service mappings.
  • 2016GCE & AWS Ingress Controllers — Cloud-specific controllers emerged, wiring Ingress resources to ALB/GLB. The vendor-per-annotation problem began here.
  • 2018Traefik 1.x, Contour, HAProxy — Alternative controllers proliferated. Annotations diverged badly — nginx.ingress.kubernetes.io/* vs traefik/* made resources non-portable.
  • 2019Kubernetes 1.14 — IngressClass concept proposed to disambiguate multiple controllers in one cluster.
  • 2020Kubernetes 1.18 — IngressClass resource added, pathType (Exact, Prefix, ImplementationSpecific) formalised. Still in beta.
  • 2021Kubernetes 1.19 — Ingress graduates to GA (v1). networking.k8s.io/v1 API. Old extensions/v1beta1 deprecated.
  • 2022Gateway API v0.5 — The SIG-Network community begins replacing Ingress with the more expressive, role-oriented Gateway API (HTTPRoute, GRPCRoute, TCPRoute).
  • 2024Gateway API v1.0 GA — HTTPRoute and Gateway reach stable. Industry moves toward Gateway API for new deployments; Ingress remains widely used for existing workloads.

Ingress started as a minimal HTTP routing abstraction but was stretched beyond its design by annotation bloat. The Kubernetes community's answer is the Gateway API, which provides a richer, role-separated, and extensible model. Ingress won't disappear soon — millions of clusters depend on it — but new architectures increasingly prefer Gateway API.

Core Concepts — Basic 🟢

1. The Ingress Resource

An Ingress is a YAML manifest you apply to your cluster. It contains rules mapping hostnames and URL paths to Kubernetes Services. By itself it does nothing — it needs an Ingress Controller to watch and act on it.

YAML
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  namespace: production
spec:
  ingressClassName: nginx
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80

2. The Ingress Controller

The controller is a Pod (or set of Pods) running in your cluster that watches Ingress resources via the Kubernetes API and configures an actual reverse proxy (NGINX, Traefik, HAProxy, Envoy, etc.) to match the declared rules. Without a controller, Ingress resources are ignored.

3. IngressClass

When multiple controllers run in one cluster (NGINX for production, Traefik for staging), ingressClassName tells each Ingress which controller should handle it. The controller is configured with a corresponding IngressClass resource.

YAML
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/ingress-nginx

5. TLS Termination (Basic)

Add a tls section referencing a Secret that holds your certificate and key. The controller handles the HTTPS handshake; traffic inside the cluster travels as plain HTTP to the backend Service.

YAML
spec:
  tls:
  - hosts:
    - app.example.com
    secretName: app-tls-secret   # kubectl create secret tls ...
  rules:
  - host: app.example.com
    # ... rules as normal
💡

Use cert-manager with a Let's Encrypt ClusterIssuer and the annotation cert-manager.io/cluster-issuer: letsencrypt-prod to fully automate certificate issuance and renewal — zero manual cert management.

Core Concepts — Intermediate 🟡

1. Annotations for Advanced Behaviour

Ingress annotations are the escape hatch for controller-specific features. They're powerful but non-portable — an annotation for NGINX won't work on Traefik. This is the primary driver behind the Gateway API migration.

YAML
metadata:
  annotations:
    # Rewrite /app/foo → /foo before forwarding to backend
    nginx.ingress.kubernetes.io/rewrite-target: /$2

    # Rate limiting: 10 req/s per IP
    nginx.ingress.kubernetes.io/limit-rps: "10"

    # Enable CORS
    nginx.ingress.kubernetes.io/enable-cors: "true"

    # Force HTTPS redirect
    nginx.ingress.kubernetes.io/ssl-redirect: "true"

    # Custom upstream timeout
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"

    # Use sticky sessions (cookie-based)
    nginx.ingress.kubernetes.io/affinity: "cookie"
    nginx.ingress.kubernetes.io/session-cookie-name: "INGRESSCOOKIE"

2. URL Rewriting

A common pattern: expose services under a path prefix externally (/api/orders) but the backend expects just /orders. The capture group in the path regex + rewrite-target annotation strips the prefix.

YAML
metadata:
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /orders(/|$)(.*)
        pathType: ImplementationSpecific
        backend:
          service:
            name: orders-service
            port:
              number: 8080
# Request: GET /orders/123  → forwarded as GET /123
⚠️

Gotcha: Forgetting capture groups in path rewrites causes the entire path to be rewritten to /, breaking all non-root requests. Always test rewrite rules with a staging Ingress first.

3. Name-based Virtual Hosting

Multiple hostnames on the same Ingress object (or multiple Ingress objects sharing a controller) route to completely different services — just like Apache VirtualHosts.

YAML
spec:
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service: { name: frontend, port: { number: 80 } }
  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service: { name: api-gateway, port: { number: 8080 } }

4. Default Backend

When no rule matches a request (unknown hostname, unknown path), the Ingress controller returns a response from the default backend — typically a simple 404/503 page. You can customise it to return branded error pages.

YAML
spec:
  defaultBackend:          # catches all unmatched requests
    service:
      name: custom-error-page
      port:
        number: 80

5. cert-manager Integration

cert-manager watches Ingress objects with special annotations and automatically provisions, stores, and renews TLS certificates from Let's Encrypt (or any ACME provider). Zero manual certificate management.

YAML
metadata:
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
  - hosts: [app.example.com]
    secretName: app-tls   # cert-manager creates and populates this Secret
💡

cert-manager uses HTTP-01 or DNS-01 ACME challenges. HTTP-01 works for public clusters. DNS-01 is needed for private clusters or wildcard certs — requires a DNS provider integration (Route53, Cloudflare, etc.).

Core Concepts — Advanced 🔴

1. High Availability & Horizontal Scaling of the Controller

The Ingress Controller is a critical single point of failure if run as a single replica. In production, run at least 3 replicas across different nodes using podAntiAffinity. The controller's Service is a LoadBalancer with a static IP — all replicas share it via the cloud LB's backend pool. Each controller replica independently processes NGINX/Envoy config reloads triggered by Ingress changes.

YAML
kind: Deployment
spec:
  replicas: 3
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: ingress-nginx
            topologyKey: kubernetes.io/hostname
      containers:
      - name: controller
        args:
        - --election-id=ingress-controller-leader  # leader election for config
        - --ingress-class=nginx
        resources:
          requests: { cpu: 100m, memory: 256Mi }
          limits:  { cpu: 1000m, memory: 1Gi }
💡

Trade-off: More controller replicas = better availability, but each replica independently reloads NGINX config on every Ingress change. At scale (1000+ Ingress objects), aggressive HPA on the controller can cause config reload storms. Tune --sync-period and use batch updates.

2. Canary / Traffic Splitting via Annotations

NGINX Ingress Controller supports weight-based traffic splitting through a "canary" Ingress pattern — a secondary Ingress object with nginx.ingress.kubernetes.io/canary: "true" that intercepts a percentage of requests before they reach the primary backend.

YAML
# Primary Ingress (stable — receives 90% of traffic)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-stable
spec:
  rules:
  - host: app.example.com
    http:
      paths:
      - { path: /, pathType: Prefix, backend: { service: { name: app-v1, port: { number: 80 } } } }
---
# Canary Ingress (new version — receives 10% of traffic)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-canary
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"
spec:
  rules:
  - host: app.example.com
    http:
      paths:
      - { path: /, pathType: Prefix, backend: { service: { name: app-v2, port: { number: 80 } } } }

3. WAF Integration & ModSecurity

NGINX Ingress can embed ModSecurity (Web Application Firewall) inline via a ConfigMap. This inspects every request for OWASP Top 10 attack patterns (SQLi, XSS, etc.) before forwarding. The trade-off is latency (+5–20ms per request) and false positive management.

YAML
# Enable ModSecurity globally via controller ConfigMap
kind: ConfigMap
metadata:
  name: ingress-nginx-controller
data:
  enable-modsecurity: "true"
  enable-owasp-modsecurity-crs: "true"
  modsecurity-snippet: |
    SecRuleEngine On
    SecRequestBodyAccess On
    SecAuditLog /dev/stdout
⚠️

Works in dev, breaks in prod: ModSecurity in DetectionOnly mode in dev silently allows malicious requests. Switching to SecRuleEngine On in prod often triggers false positives on legitimate API payloads (JSON bodies with special characters). Always tune rules in staging under realistic traffic before enabling enforcement.

4. Ingress at Scale — NGINX Config Reload Problem

Every time an Ingress resource changes (a new deployment updates an annotation, cert-manager rotates a cert), NGINX Ingress Controller generates a new nginx.conf and triggers a graceful reload. In large clusters (500+ Ingress objects, frequent deployments), this causes: brief connection interruptions during reload, high CPU spikes, and increased P99 latency. The fix is to use the --update-status-on-shutdown=false flag, tune --sync-rate-limit, or migrate to Envoy-based controllers (Contour, Emissary) that support hot-reload without process restart.

Internal Walkthrough — How NGINX Ingress Processes a Request

  1. Client DNS resolution: Client resolves app.example.com → cloud LB external IP (static, assigned to the Ingress Controller's Service of type LoadBalancer).
  2. Cloud LB → Controller Pod: The cloud LB selects one of the N running Ingress Controller Pods and forwards the TCP connection.
  3. TLS handshake: If HTTPS, the controller performs TLS termination using the certificate from the referenced Secret. The connection to the backend is typically plain HTTP.
  4. NGINX config lookup: NGINX evaluates its server blocks (one per hostname) then location blocks (one per path rule) in longest-prefix order to find a match.
  5. Rewrite & header manipulation: Annotations are applied — path rewrites, header injections (X-Forwarded-For, X-Real-IP), rate limiting checks.
  6. Upstream selection: NGINX uses its upstream configuration (built from the Service's Endpoints, refreshed without reload via the Lua balancer_by_lua module) to pick a backend Pod IP.
  7. Request forwarded: NGINX opens a connection to the Pod directly (bypassing kube-proxy in modern controllers using EndpointSlices) and proxies the request.
  8. Response returned: The backend response travels back through NGINX (where response headers may be manipulated) → cloud LB → client.

Key Features

  • Host-based routing — Route api.example.com and app.example.com to different services on the same IP and port.
  • Path-based routing — Direct /api/*, /auth/*, /static/* to independent backend services.
  • TLS termination — Handle HTTPS at the ingress boundary; backend services can run plain HTTP internally.
  • Automatic certificate management — cert-manager integration for Let's Encrypt; auto-renewal before expiry.
  • URL rewriting — Strip or transform path prefixes before forwarding to backends.
  • Rate limiting — Per-IP or per-source rate limiting at the ingress layer, before requests reach services.
  • Session affinity — Cookie-based sticky sessions for stateful backends.
  • Canary deployments — Weight-based or header-based traffic splitting between service versions.
  • Basic authentication — HTTP Basic Auth enforced at the ingress, protecting internal services without app changes.
  • Custom error pages — Branded 404/503 responses via a default backend service.
  • CORS headers — Inject Cross-Origin Resource Sharing headers centrally, without modifying each service.
  • WAF integration — ModSecurity or external WAF (AWS WAF, Cloudflare) inline protection.
10

Key Benefits

  • ✅ Cost reduction — One cloud load balancer fronts hundreds of services instead of one LB per service — a major saving in cloud bills at scale.
  • ✅ Centralised TLS management — Certificates managed in one place with cert-manager; no per-service SSL renewal burden.
  • ✅ Decoupled routing from app code — Services don't need to know their external URLs; routing is infra configuration.
  • ✅ Declarative configuration — Ingress rules are YAML, stored in Git, version-controlled, reviewed, and deployed via CI/CD like any other manifest.
  • ✅ Namespace isolation — Each team manages their own Ingress resources in their namespace; the platform team owns the controller.
  • ✅ Progressive delivery — Canary annotations enable safe rollouts without additional tooling like a service mesh.
  • ✅ Observability built-in — NGINX metrics (requests/s, latency, upstream response codes) exported to Prometheus out of the box.

Key Challenges & Pitfalls

  • ⚠️ Annotation lock-in — Using 20 NGINX-specific annotations makes your Ingress manifests non-portable. Switching controllers requires rewriting all annotations. Gateway API solves this.
  • ⚠️ Config reload disruption — Every Ingress change triggers an NGINX reload. In busy clusters, this causes brief request drops. Tune --sync-rate-limit and consider Envoy-based controllers for hot-reload.
  • ⚠️ Path ordering surprises — More specific paths must come before less specific ones in some controllers. /api/v2 must be listed before /api, or it may never match. Always use pathType: Exact when precision matters.
  • ⚠️ Single-controller SPOF — Running one controller replica for all namespaces is a cluster-wide single point of failure. Run 3+ replicas with pod anti-affinity across nodes.
  • ⚠️ Wildcard TLS and cert mismatch — A TLS secret for *.example.com won't automatically match two-level subdomains (api.v2.example.com). Cert-manager wildcard certs require DNS-01 challenge and careful domain scoping.
  • ⚠️ Works in dev, breaks in prod — missing proxy protocol: When a cloud LB sits in front of the Ingress Controller without proxy protocol configured, X-Forwarded-For contains the LB's IP, not the real client IP. Rate limiting by IP and geo-blocking then apply to the LB's address, not the attacker's.
  • ⚠️ Ingress resource proliferation — Teams creating hundreds of Ingress objects without cleanup leads to stale rules, confusing debugging, and controller memory growth. Enforce naming conventions and use admission webhooks to validate.

Best Practices

🟢 Beginner

  • Always set ingressClassName explicitly — don't rely on the default class annotation to avoid surprises when multiple controllers are present.
  • Use pathType: Prefix for most routes and pathType: Exact when you need to match only one specific path.
  • Always configure TLS, even in staging — use cert-manager with Let's Encrypt staging issuer to avoid rate limits.
  • Test ingress rules with curl -H "Host: app.example.com" http://<controller-ip>/path before adding DNS.

🟡 Intermediate

  • Keep Ingress annotations minimal — if you need more than 5 annotations, consider whether a service mesh or Gateway API is a better fit.
  • Use one Ingress per application (not one per namespace) for easier debugging and cleaner RBAC.
  • Configure controller resource requests/limits and HPA — the controller is on the critical path for all external traffic.
  • Integrate Prometheus metrics from NGINX controller into your SLO dashboards — track nginx_ingress_controller_requests by status code per ingress.

🔴 Advanced / Production

  • Run 3+ controller replicas with podAntiAffinity spread across nodes and availability zones; use PodDisruptionBudget to prevent all replicas being evicted during node drains.
  • Enable --enable-ssl-passthrough only if truly needed — it disables host-based routing for that traffic and increases complexity.
  • For clusters with 200+ Ingress objects, measure and tune NGINX reload frequency — batch changes where possible and consider Envoy-based controllers (Contour, Emissary) for zero-downtime hot-reload.
  • Implement admission webhooks (e.g. conftest, OPA Gatekeeper) to validate Ingress manifests — catch duplicate hosts, missing TLS, or banned annotations before they hit the cluster.
  • Plan your Gateway API migration now — evaluate HTTPRoute parity with your current annotations and pilot one namespace before full migration.

Key Trends & Future Outlook

  • Gateway API superseding Ingress — HTTPRoute, GRPCRoute, and TCPRoute reached GA in 2023–2024. All major controllers (NGINX, Contour, Istio, Cilium) now support it. Ingress remains in maintenance mode; Gateway API is the future. Maturity: GA, rapidly growing adoption.
  • eBPF-based controllers (Cilium) — Cilium's Ingress and Gateway API implementation operates at the kernel level via eBPF, bypassing kube-proxy entirely. This delivers lower latency, higher throughput, and richer observability than NGINX-based controllers. Maturity: Production-ready, growing fast.
  • AI-driven traffic management — APM vendors (Datadog, Dynatrace) integrate with Ingress metrics to auto-suggest canary rollback, detect anomalous traffic patterns, and recommend rate limit tuning. Maturity: Early production in enterprise tools.
  • Zero-trust ingress — Combining Ingress with SPIFFE/SPIRE for workload identity, and requiring mTLS even for north-south traffic. Cloud providers offering native zero-trust ingress options (GKE Workload Identity + Cloud Armor). Maturity: Growing in security-conscious enterprises.
  • Multi-cluster ingress — Platforms routing external traffic across multiple clusters (GKE Multi-cluster Ingress, Submariner) for active-active geo-redundancy, without a global CDN dependency. Maturity: Production-ready on GKE/EKS.

Interview Questions 🎯

Senior-level questions for architects, leads, and platform engineers.

🟢 Conceptual & Fundamentals

Q: What is the difference between a Kubernetes Service of type LoadBalancer and an Ingress? When would you use each?
What a great answer covers: LoadBalancer provisions one cloud LB per Service (Layer 4, TCP/UDP). Ingress is Layer 7 (HTTP/HTTPS), routing many services through one cloud LB based on host/path. Use LoadBalancer for non-HTTP protocols (gRPC raw TCP, databases), Ingress for HTTP services. Discuss cost implications — 1 LB vs N LBs at scale.
Q: What happens if you apply an Ingress resource but no Ingress Controller is installed in the cluster?
What a great answer covers: Nothing happens — the resource is stored in etcd but no controller is watching it. No routing occurs, no error is returned. This is a common gotcha for newcomers. The Ingress controller is the "brain" that reads the resource and configures the actual proxy. Mention IngressClass as the mechanism to match resource to controller.
Q: Explain pathType: Exact vs Prefix. Give an example where using the wrong one causes a production incident.
What a great answer covers: Exact matches only that literal path. Prefix matches path and all sub-paths. Production incident: path: /admin, pathType: Prefix on a public-facing Ingress accidentally exposes /admin/users, /admin/config etc. A health check endpoint at /actuator/health accessible due to Prefix on /actuator. Always use Exact for sensitive paths.
Q: How does cert-manager automate TLS certificate management with Ingress?
What a great answer covers: cert-manager watches Ingress resources with the annotation cert-manager.io/cluster-issuer. It creates a Certificate resource, challenges the ACME server (HTTP-01 by creating a temporary Ingress route, or DNS-01 by creating a DNS TXT record), receives the cert, stores it in the referenced Secret, and renews it 30 days before expiry. Discuss HTTP-01 vs DNS-01 trade-offs and wildcard cert requirements.

🟡 Design & Architecture

Q: You're designing a platform for 50 dev teams sharing one Kubernetes cluster. How do you architect Ingress management to give teams autonomy while maintaining platform control?
What a great answer covers: Teams own Ingress resources in their namespaces. Platform team owns the controller and IngressClass. Use RBAC to restrict which annotations teams can use. OPA Gatekeeper admission webhooks enforce: no duplicate hostnames across namespaces, mandatory TLS, approved domain suffixes only. Use ExternalDNS for automatic DNS. Consider namespace-scoped controllers for full isolation. Discuss Gateway API's role-separation (infrastructure/application/route roles) as the future state.
Q: Walk me through how you'd implement a blue/green deployment using Kubernetes Ingress.
What a great answer covers: Two Deployments (blue, v1; green, v2) each behind their own Service. Single Ingress initially points 100% to blue. Canary Ingress with weight annotation gradually shifts traffic to green (10% → 50% → 100%). Monitor error rates and latency during each step. On completion, update primary Ingress to green service, remove canary Ingress. Discuss Argo Rollouts as automation for this process. Contrast with a service mesh approach.
Q: How would you design Ingress for a multi-region active-active architecture?
What a great answer covers: Global load balancer (AWS Global Accelerator / GCP GCLB / Cloudflare) routes to regional clusters based on latency/health. Each regional cluster has its own Ingress Controller. DNS failover with health checks detects regional Ingress Controller outage. Multi-cluster Ingress (GKE) as a managed alternative. Discuss session affinity challenges, data replication requirements, and the trade-off between active-active complexity and RTO/RPO goals.
Q: When would you choose Gateway API over Ingress for a new project?
What a great answer covers: Gateway API when: multi-tenant platform (role separation is built-in), need non-HTTP protocols (TCPRoute, UDPRoute, GRPCRoute), want to avoid annotation lock-in, need weighted traffic splitting natively, team is greenfield and not carrying Ingress migration debt. Ingress when: simpler cluster, existing annotation-based config is working, team is familiar, controller doesn't fully support Gateway API yet. Mention all major controllers now have Gateway API support.

🔴 Advanced, Internals & Trade-offs

Q: Explain the NGINX config reload problem in detail. What causes it, what are the symptoms, and how do you mitigate it?
What a great answer covers: Every Ingress change → controller regenerates nginx.conf → sends SIGHUP → NGINX graceful reload. During reload: new workers start, old workers drain. With 500+ Ingress objects and frequent deploys, this can happen every few seconds. Symptoms: P99 latency spikes, brief 502s on connections hitting dying workers. Mitigation: ingress-nginx Lua-based upstream updates avoid reload for endpoint changes; tune --sync-rate-limit; batch Ingress updates in CI; consider Envoy-based controllers (Contour) that use xDS API for hot-reload without process restart.
Q: How does the NGINX Ingress Controller update backend endpoints without a full config reload?
What a great answer covers: NGINX Ingress uses a Lua module (lua-resty-balancer) to maintain an in-memory upstream table. When EndpointSlices change (pods scale up/down), the controller pushes new Pod IPs to the Lua module via a Unix socket — without regenerating nginx.conf or triggering a reload. Only structural changes (new Ingress rules, new TLS certs) require a full reload. This is a significant scalability improvement over older versions that reloaded on every endpoint change.
Q: A client's IP address appears as the cluster's load balancer IP instead of the real client IP in application logs. Diagnose and fix.
What a great answer covers: Root cause: cloud LB not forwarding the client IP via X-Forwarded-For or PROXY protocol. Fix options: (1) Configure cloud LB to add X-Forwarded-For — set use-forwarded-headers: "true" in NGINX ConfigMap. (2) Enable PROXY protocol on the cloud LB and NGINX controller (use-proxy-protocol: "true") for Layer 4 passthrough of client IP. (3) Use ExternalTrafficPolicy: Local on the controller Service — preserves source IP but sacrifices even load distribution. Discuss security implications of trusting X-Forwarded-For without validation.
Q: How would you debug a 502 Bad Gateway returned from an Ingress for a specific path?
What a great answer covers: Systematic approach: (1) Check Ingress controller logs — kubectl logs -n ingress-nginx deploy/ingress-nginx-controller. (2) Check if the backend Service exists and has healthy Endpoints — kubectl get endpoints <service>. (3) Verify pathType and path regex are correct — test with curl -H "Host:" directly against controller Pod IP. (4) Check if the backend Pod is actually ready — readiness probe passing? (5) Verify port numbers match between Ingress, Service, and container. (6) Check for upstream timeout annotations if backend is slow to respond.

💡 Scenario / Problem-Solving

Q: "You're seeing a sudden 10× spike in 504 Gateway Timeout errors on your Ingress. Walk me through your incident response."
What a great answer covers: Check Grafana — which Ingress/service, which path, which region. Check controller pod metrics — is the controller itself healthy or CPU-saturated? Check backend pod health and readiness. Check if a recent deployment changed resource limits or config. Look for upstream connection pool exhaustion. Check for a cloud LB health check failure causing traffic to re-route. Check for cert-manager cert renewal triggering a reload storm. Mitigation: scale backend, increase proxy-read-timeout annotation, implement circuit breaker upstream.
Q: "Your team wants to migrate from NGINX Ingress to Gateway API HTTPRoute with zero downtime. How do you plan and execute this?"
What a great answer covers: Start with one low-traffic namespace as a pilot. Install Gateway API CRDs alongside existing Ingress. Deploy a controller that supports both (NGINX, Contour). Create HTTPRoute equivalents for existing Ingress rules. Test with shadow traffic or canary. Migrate one namespace at a time, validating with smoke tests. Handle annotation equivalents (rate limiting → RequestRate filter, auth → ReferenceGrant). Plan fallback: keep Ingress objects until HTTPRoute is proven. Update CI/CD templates last. Total migration: 2–4 sprints depending on annotation complexity.
Q: "A developer says their new service is returning 404 from the Ingress even though the path rule looks correct. Debug it."
What a great answer covers: Verify IngressClass is set and matches the running controller. Check if the Ingress is in the same namespace as the Service (backend service references are namespace-scoped). Confirm pathType — Exact may not match sub-paths as expected. Check if a more specific rule in another Ingress is capturing the request first (host+path ordering). Exec into controller pod and inspect nginx.conf to see if the rule was actually generated. Test curl directly against the controller Pod IP bypassing DNS. Check backend Service selector matches Pod labels.
